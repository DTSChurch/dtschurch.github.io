Workflow

Connection request lifecycle

Serve System can create Rock Connection Requests automatically during sign-up when the selected role has a ConnectionOpportunityId. This allows new volunteers to enter Rock’s connection follow-up process without requiring a separate manual handoff.

Optional per role Duplicate prevention Family-member aware Campus-based connector assignment

Role Configuration

Setting Purpose
Connection Opportunity Enables connection request creation for the role. If blank, sign-ups do not generate requests.
DataView Optional filter that restricts which people receive a connection request while still allowing sign-up.

Creation Flow

1

Collect all selected shifts first

The service gathers every selected shift time before individual sign-up records are processed so the request comment contains the full schedule.

2

Check whether the role is connection-enabled

If the role has no ConnectionOpportunityId, Serve continues with sign-up but does not create or attach any connection request.

3

Look for an existing active request

The system queries for an active request matching the volunteer and the role’s connection opportunity.

4

Evaluate the optional DataView filter

If no active request exists, the role-level DataView is checked. People who fail the filter still get signed up, but their connection request is skipped.

5

Create and populate the request

Serve writes the request fields, aggregates the full comment, and assigns the default connector based on the selected or inferred campus.

6

Attach the request to all related sign-ups

All sign-up records for that person in the current operation point at the single request through ConnectionRequestId.

Family Member Handling

Separate Requests

Family members receive their own connection requests rather than sharing the primary person’s request.

Primary Person Attribution

The generated comment includes the primary person name so coordinators know who completed the sign-up.

Independent Filtering

The same DataView logic runs for each family member independently.

Campus Fallback

If no campus was selected explicitly, the family member’s primary campus is used for defaults and connector assignment.

Duplicate Prevention And Linkage

Duplicate Prevention

Before creating a new request, Serve looks for any active request where PersonAliasId, ConnectionOpportunityId, and ConnectionState == Active all match. If one exists, the sign-up reuses it.

Sign-Up Linkage

ServeOpportunityInstanceSignUp.ConnectionRequestId is nullable. It is populated when a request was created or reused and remains null when the role is not configured or the DataView filter excludes the person.

Sign-Up Cancellation Behavior

DeactivateConnectionRequestIfOrphaned only deactivates the request when no other active sign-ups still point to it. When it does deactivate, the request is marked inactive, an activity note is written, and an auto-inactivate status is applied if one exists.

Troubleshooting

Connection Request Not Created

Check the role’s ConnectionOpportunityId, verify the person passes any configured DataView, and confirm an active request does not already exist for the same person and opportunity.

Only One Shift In Comments

If comments show only one shift, verify the running plugin version includes the aggregated shift-time fix before request creation.

Wrong Connector

Review campus-specific default connectors on the Rock Connection Opportunity. The plugin calls GetDefaultConnectorPersonAliasId(campusId).

Family Member Missing A Request

Verify the family member passes the DataView filter and has a valid PrimaryAliasId; their request is evaluated independently from the primary person.
