Scheduling

Shift times and expiration behavior

Serve shift timing is resolved from the most specific matching override record, while sign-up cutoff behavior falls back from shift-level expiration to the instance schedule expiration. The current model now includes exact-role, role-category, campus, all-role, schedule-default, and instance-default behavior layered in a fixed precedence order.

Shift Time Resolution Order

1

Schedule + Role + Campus

Most specific override for a particular schedule, role, and campus combination.

2

Schedule + Role

Role override for one schedule across campuses.

3

Role + Campus

Role default for a campus across schedules when such records exist.

4

Role

Exact-role fallback when schedule and campus-specific overrides are not present.

5

Schedule + Role Category + Campus

Category-scoped override for one schedule and campus when no exact-role match exists.

6

Schedule + Role Category

Category override for one schedule across campuses.

7

Role Category + Campus

Category default for a campus across schedules.

8

Role Category

Category-wide fallback when no more specific role or category scope matches.

9

Schedule + Campus

Schedule-level campus override when role and category detail are absent.

10

Campus

Campus-only fallback that applies across roles and categories.

11

Schedule Default

Default shift time assigned to the instance schedule.

12

Instance Default

Fallback default stored at the broader instance level.

13

Base iCal Schedule Time

Final fallback to the underlying schedule definition when no override record matches.

Admin Surface Ownership

Serve Opportunity Instance Detail

Owns instance schedules, schedule default shift times, schedule expirations, all-schedules scoped defaults, and schedule-scoped overrides.

Instance Detail In Campus Mode

Also manages exact-role, role-category, and all-role campus variants when the instance is partitioned by campus.

Role Schedule Configurator

Owns role counts per schedule cell, schedule/role shift-time edits in the grid, and per-shift expiration updates for the selected campus context.

Current Editing Model

All-Schedules Scoped Defaults

Instance detail edits the broad defaults that apply across schedules for exact roles, role categories, or all roles.

Schedule Defaults

Base schedule times are edited in instance detail and serve as a later fallback when more specific records are absent.

Schedule + Role

Exact-role overrides can be edited in both instance detail and the configurator.

Schedule + Role Category

Role-category overrides are edited in instance detail and apply when no exact-role override wins first.

Campus-Scoped Overrides

Schedule + role + campus and schedule + all roles + campus overrides are available in instance detail when the instance is campus-partitioned.

Configurator Context

The configurator always works in the currently selected campus context when the instance is partitioned by campus.

The resolution service also understands additional shapes such as Role + Campus, Role Category + Campus, Campus-only, and Instance Default records even though the current admin UI does not foreground every one of those shapes equally.

Expiration Precedence

1

Shift expiration is checked first

If a specific shift has its own expiration, that timestamp decides whether sign-up is still open.

2

Schedule expiration is the fallback

If the shift does not have an explicit expiration, Serve falls back to the instance schedule expiration.

3

No expiration means sign-up stays open

If neither layer defines an expiration, the shift remains available unless other constraints apply.

Where To Edit Expirations

Instance Detail

Use this when the whole schedule for the instance should close at the same time, such as every Saturday 5 PM shift closing Friday at noon.

Role Schedule Configurator

Use this when one concrete shift needs its own expiration. The expirations tab shows the schedule-level value as the starting fallback, but saving there writes a shift-level expiration.

Campus Filtering Notes

When an instance is partitioned by campus, instance detail can save overrides for all campuses or one specific campus. The configurator always works in the selected campus context and does not currently expose a separate no-campus editing mode for partitioned instances.

Practical Examples

Schedule Default Only

If Sunday 9 AM has a default of 8:00 AM to 9:00 AM and no role override exists, every role inherits that time range.

Role Override For All Campuses

If Greeter has an all-schedules default of 7:30 AM to 9:00 AM, Greeters use that while other roles still use the schedule default.

Campus-Specific Role Override

If Northwest Greeters have a campus-specific default of 7:15 AM to 9:00 AM, that beats the all-campus Greeter default for Northwest only.

Expiration Fallback

If the schedule expires Friday at noon and one specific shift expires Thursday at 5:00 PM, that one shift closes earlier while the rest follow the schedule rule.

Category Default

If the Guest Services category default is 7:15 AM to 9:00 AM and Greeter belongs to Guest Services, Greeter inherits that category default until a more specific role or schedule override exists.

All-Role Campus Default

If Northwest has a schedule-scoped all-role override of 7:45 AM to 9:00 AM, every role at Northwest inherits it unless a role-specific or role-category override is more specific.
