Automation

SyncPlanningCenter Job

com.dts.RockLinkPlanningCenter.Jobs.SyncPlanningCenter is the single Rock Job that drives the sync. It reads the singleton settings record, walks every active group/team mapping in EntityMap, delegates the actual work to PlanningCenterSyncService, and writes a SyncLog row for every run.

Configuration

The job is registered as a Rock Service Job. Schedule and run history are managed from Admin Tools → System Settings → Jobs Administration, and the Settings block links directly to it via the “Job Schedule” shortcut. The block’s Run Sync Now button enqueues a RunJobNowTransaction against this job.

Job Attributes

AttributeTypeDefaultDescription
Command Timeout Integer (seconds) 180 Maximum time the SQL command may run for any operation in the sync. Applied to RockContext.Database.CommandTimeout.

Inputs Read From Settings

  • ApiApplicationId — PCO Application ID.
  • ApiSecret — encrypted secret; decrypted via Rock.Security.Encryption.DecryptString at run start.
  • DefaultConnectionStatusValueId — applied to newly created people.
  • OverwriteWithPlanningCenter — conflict policy.

If credentials are missing, the job aborts before any PCO calls, writes a Failed SyncLog row with the reason, and throws so the failure is visible in the Rock Jobs grid.

Run Lifecycle

  1. Open a single RockContext with the configured command timeout.
  2. Insert a SyncLog row with Status = "Running", SyncPhase = "All", and StartDateTime = now.
  3. Load settings; abort to Failed if credentials are unset.
  4. Decrypt the API secret.
  5. Query EntityMap for active rows where EntityTypeId = Group and PlanningCenterEntityType IN ("Group", "Team"). Exit cleanly to Completed if there are none.
  6. Construct one PlanningCenterSyncService for the run.
  7. For each mapping, dispatch to SyncPlanningCenterGroup (PCO Group) or SyncServiceTeam (PCO Team). The service handles person upsert, member upsert, and role mapping. Status messages flow back to UpdateLastStatusMessage so the Jobs grid shows live progress.
  8. Aggregate the per-mapping counters back onto the job’s totals (people / groups / members — added / updated / skipped / errored).
  9. Build the Result string (multi-line summary, plus the first 10 error messages and an “and N more” tail when applicable).
  10. Update the SyncLog row with final counts and either Completed or CompletedWithErrors.

What Each Mapping Does

PCO Group

SyncPlanningCenterGroup

Fetches the group from PCO Groups, syncs every member as a Person (creating or updating per the overwrite policy), then upserts each membership into the linked Rock Group.

PCO Service Team

SyncServiceTeam

Fetches the team and its team members from PCO Services, syncs each as a Person, then upserts each membership into the linked Rock Group. PCO team-position → Rock GroupRole mapping (stored in EntityMap.AdditionalData) is applied if configured.

Cross-cutting

Person Resolution

Linked people are resolved through the PersonAlias EntityMap. Brand-new people fall through to FindPerson for first-time matching by name/email/phone, then a new EntityMap row is written so the link survives future runs.

Counters Tracked Per Run

The job aggregates three buckets, each with four sub-counts:

PeopleNew Person rows createdExisting Person rows touchedLinked Person rows skipped (e.g. nothing changed under non-overwrite policy)Person upsert failures
GroupsReserved (groups are pre-mapped, not created)Incremented per mapping processedMappings skippedMapping-level failures

The SyncLog row stores collapsed totals: RecordsCreated, RecordsUpdated, RecordsSkipped, RecordsErrored, and a derived RecordsProcessed = created + updated + skipped (across all three buckets).

Error Handling

  • Per-record errors are caught inside the sync service, recorded onto the relevant counter, and pushed onto an in-memory ErrorMessages list. They do not abort the run.
  • The first 10 errors are echoed into the job’s Result string. The first 50 are persisted into SyncLog.ErrorDetails.
  • Per-mapping fatal errors mark the mapping’s SyncStatus = "Error" with the exception message in EntityMap.ErrorMessage.
  • Run-level failures (e.g. credentials missing, fatal exception bubbling out of the loop) write Status = "Failed" on the SyncLog and rethrow so Rock’s job runner records the failure.
  • End-of-run status is CompletedWithErrors if any per-record errors were collected, otherwise Completed.

Manual Trigger From The Settings Block

The Settings block exposes Run Sync Now for ad-hoc runs. Behavior:

  • Looks up the service job by class name (com.dts.RockLinkPlanningCenter.Jobs.SyncPlanningCenter).
  • Refuses to enqueue if the job’s LastStatus is Running.
  • Otherwise calls new RunJobNowTransaction(syncJob.Id).Enqueue(), which schedules the run on Rock’s transaction queue. The block returns immediately; the UI polls RefreshSyncStatus for live updates.

Scope & Non-Goals

  • The job only processes EntityMap rows scoped to Rock Groups linked to PCO Groups or Service Teams. Person and Campus mappings are created and consumed during a run; they are not iterated as a top-level phase.
  • There is no incremental-only mode in v1 — each active mapping is fully reconciled on every run.
  • The job does not pull attendance or volunteer schedules.
