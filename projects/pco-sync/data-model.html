Concepts

Data Model

The plugin keeps three pieces of information in Rock: the links between Rock records and their Planning Center counterparts, the history of every sync run, and the connection settings for talking to Planning Center. This page describes each in plain language so you know what the sync is tracking on your behalf.

Links

Entity mappings

One entry per linked record — this person in Rock matches that person in PCO, this Rock Group matches that PCO team, and so on.

History

Sync log

One entry per sync run, with totals, duration, status, and any errors.

Connection

Settings

A single record holding the Planning Center credentials and your sync policy.

Entity Mappings

The link between a Rock record and its Planning Center counterpart. There’s one mapping for each pairing the sync knows about.

Types of mappings

TypeLinksCreated By
GroupA Rock Group to a Planning Center Group.You, on the Groups page.
Service TeamA Rock Group to a Planning Center Service Team.You, on the Groups page.
CampusA Rock Campus to a Planning Center Campus.You, on the Settings page.
PersonA Rock Person to a Planning Center Person.The sync, automatically.
Group MemberA Rock group membership to its Planning Center counterpart.The sync, automatically.

What each mapping remembers

  • The Planning Center side of the link (the type and the PCO ID).
  • The cached display name from Planning Center, so the admin UI doesn’t have to refetch it.
  • The Rock record on the other end of the link.
  • For Service Team mappings, the position-to-role mapping you’ve configured.
  • The most recent successful sync date for this mapping.
  • The current sync status — e.g. Synced, Mapped, Syncing, Error, or Pending.
  • The last error message, if the most recent attempt failed.
  • An active flag — only active mappings are processed by the sync, so you can pause a mapping without losing the link.

Person mappings and merges

Person mappings are stored against a person’s Rock aliases rather than the person record directly. The practical upshot: if you merge two people in Rock, the Planning Center link automatically follows the surviving record. You don’t need to reconnect anything by hand.

Sync History

One entry is written for every sync run. The Sync History view reads from this list, newest first.

What each run records

  • The Rock job that triggered the run (when applicable).
  • The phase being synced — in v1, this is always All.
  • Start and end date/time. Duration is derived from the difference.
  • Final status — Running, Completed, Completed With Errors, or Failed.
  • Records processed, created, updated, skipped, and errored — totals across people, groups, and group members.
  • Error details. Up to 50 individual error messages are kept, plus the fatal error message for failed runs.

Run history is never overwritten — each run produces its own entry, so you have a permanent audit trail. There is no automatic pruning in v1.

Settings

A single record holding the Planning Center connection and your sync policy. The Settings page reads and writes this record — you don’t need to touch it directly.

What it holds

  • Your Planning Center API Application ID.
  • Your Planning Center API Secret, encrypted at rest. Once saved, it’s never displayed in plain text again.
  • The default Connection Status for brand-new people.
  • The overwrite policy — whether Planning Center data overwrites Rock fields on linked records, or only fills in blanks.

Putting It Together

The three pieces work as follows during a normal sync run:

  1. The sync job reads Settings to get the credentials and the overwrite policy.
  2. It looks up the active entity mappings you’ve configured, fetches each one’s latest data from Planning Center, and creates or updates the matching Rock records.
  3. For brand-new people it encounters in PCO, it creates the Rock person and writes a new Person mapping so the link survives future runs.
  4. Each mapping’s last sync date and status are updated as it’s processed.
  5. When the run finishes, a sync history entry is written with the final status, totals, and any error details.
